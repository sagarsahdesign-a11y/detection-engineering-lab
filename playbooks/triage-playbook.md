# SOC Incident Response & Triage Playbook

This document provides structured procedures for Tier 1 and Tier 2 Security Operations Center (SOC) analysts to triage, investigate, and contain high-severity alerts generated by the Detection Engineering Lab.

---

## Playbook 1: OS Credential Dumping (T1003.001)
* **Wazuh Rule ID:** `100103` (LSASS Process Access), `1001031` (rundll32 comsvcs.dll)
* **Severity:** Critical (Level 13/14)

### 1. Initial Triage (Determine if True Positive)
Assess the process initiating the memory access:
1. **Identify the Source Image (`SourceImage`):** What executable requested a handle to `lsass.exe`?
   * *Suspect binaries:* `procdump.exe`, `mimikatz.exe`, `sekurlsa`, `rundll32.exe` invoking `comsvcs.dll`, or unknown/unsigned binaries in `C:\Users\Public` or `C:\Windows\Temp`.
   * *Legitimate binaries:* `WmiPrvSE.exe` (WMI Provider Host), `taskmgr.exe` (Task Manager - if an admin manually created a dump, still requires verification), anti-virus or EDR agents.
2. **Examine the Granted Access Mask (`GrantedAccess`):**
   * Access mask `0x1010` or `0x1410` (query information and read memory) is highly indicative of dumping tools.
   * Access mask `0x1fffff` indicates full process access.

### 2. Deep-Dive Investigation
* **Process Ancestry:** Check the parent process of the dumping utility. Was it spawned by `cmd.exe` or `powershell.exe`? Trace the process tree back to the initial entry point (e.g., did a web server or user email client spawn cmd?).
* **Command Line Analysis:** Look at the command parameters.
   * *Example:* `procdump.exe -ma lsass.exe` or `rundll32.exe C:\Windows\system32\comsvcs.dll, MiniDump 624 ...`
* **File Integrity:** Inspect the output dump file (usually ending in `.dmp` or `.tmp` in `C:\Windows\Temp` or `C:\Users\Public`). Check if this file was subsequently zipped or exfiltrated (look for file creation events or network connection events).

### 3. Containment & Remediation
1. **Host Isolation:** Immediately isolate the affected host from the network (using Wazuh's Active Response or EDR firewall isolation) to prevent credential abuse.
2. **Credential Revocation:** Force a password reset for all user accounts logged into the compromised machine at the time of the alert (especially Domain Admins).
3. **Session Termination:** Terminate active sessions associated with the compromised credentials.
4. **Eradication:** Delete the generated dump files and any associated hacking tools.

## Playbook 2: Process Injection (T1055.001)
* **Wazuh Rule ID:** `100110` (Script Host Injection), `1001101` (Suspicious cross-process Thread Injection)
* **Severity:** High (Level 12/13)

### 1. Initial Triage
Verify the process relation of Sysmon Event ID 8 (`CreateRemoteThread`):
1. **Source Process (`SourceImage`):** Which process initiated the thread creation?
   * *High Risk:* `powershell.exe`, `cmd.exe`, `mshta.exe`, `wscript.exe`, `cscript.exe`, or non-standard binaries.
2. **Target Process (`TargetImage`):** Where was the thread injected?
   * *Common targets:* `svchost.exe`, `explorer.exe`, `lsass.exe`, `spoolsv.exe` (adversaries inject here to masquerade as standard Windows services).
3. **Cross-Process Verification:** Is the Source Process different from the Target Process? (Exclude normal operating system behavior like `csrss.exe` or `winlogon.exe` calling sub-processes).

### 2. Deep-Dive Investigation
* **Network Activity:** Check if the target process (e.g., `svchost.exe` or `explorer.exe`) initiated unusual network connections immediately after the injection timestamp. This indicates a beacon/reverse shell (C2) is running inside the process space.
* **Image Loads:** Inspect if any unsigned or temporary DLLs (`C:\Users\Public\*.dll`) were loaded by the target process around the same time (Sysmon Event ID 7).
* **Command-line History:** Review what scripts were running in PowerShell (`ScriptBlockText` Event 4104) that initiated the injection.

### 3. Containment & Remediation
1. **Process Termination:** Kill the injecting process (source) and the injected process (target). *Caution: If the target is `lsass.exe` or `winlogon.exe`, killing it will crash the system; isolate the host instead.*
2. **Network Isolation:** Isolate the host to block the Command and Control channel.
3. **Forensic Analysis:** Extract a memory dump of the target process for further analysis of the payload (e.g., checking for Cobalt Strike beacons).

## Playbook 3: Windows Logon Brute Force (T1110.003)
* **Wazuh Rule ID:** `100100` (Multiple Logon Failures)
* **Severity:** High (Level 10)

### 1. Initial Triage
Identify the scale and type of the logon failure:
1. **Review Alert Frequency:** Verify that the threshold of 5+ failures within 60 seconds was met.
2. **Identify the Logon Type (`LogonType`):**
   * `3` = Network (possible network share brute-forcing or SMB/WinRM scanning)
   * `10` = RDP (Remote Desktop connection brute-forcing)
   * `2` = Interactive (physical console logon failure)
3. **Identify the Source IP (`IpAddress`):** Is the source internal or external?
   * *Internal:* Possible lateral movement attempt or a misconfigured local service account with an expired password.
   * *External:* Direct attack from internet exposure (critical threat).

### 2. Deep-Dive Investigation
* **Logon Success Correlation:** Search for a subsequent Event ID `4624` (Successful Logon) from the *same* Source IP to the *same* target username shortly after the brute force alerts. A success event indicates a successful breach!
* **User Accounts Targeted:** Is the attack targeting a single username (brute force) or scanning multiple usernames (password spraying)?
* **Check Service Account Status:** If the account is a service account (e.g., `svc-backup`), check if the failures occur continuously in exact intervals. This points to a changed password that hasn't been updated in a scheduled task or script.

### 3. Containment & Remediation
1. **Source Block:** Block the source IP address at the firewall or local host level.
2. **Account Lockout:** If not locked out automatically, temporarily disable or lock the targeted account if it is actively targeted by a successful brute-force.
3. **Password Reset:** If a success event (Event 4624) was found, immediately trigger an incident response workflow for compromise recovery:
   * Reset the user password.
   * Revoke OAuth tokens.
   * Inspect all actions taken during the session.
